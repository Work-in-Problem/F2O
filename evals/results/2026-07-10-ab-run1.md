## การตีความ

1. **Ceiling เกือบทุก metric** — Opus 4.8 ในสภาพแวดล้อมนี้ทำงานทั้งสามชิ้นได้สมบูรณ์ทั้งสอง arm
   แยกไม่ออกว่าเพราะ (ก) งานง่ายเกินไปสำหรับ Opus 4.8 หรือ (ข) Layer 0 + descriptions
   ที่ปนอยู่ใน "bare" ก็เพียงพอแล้วให้ถึงเพดาน — ข้อมูลชุดนี้แยกสองสมมติฐานนี้ไม่ได้
   (ข้อสังเกตเชิงบวก: bare arm ทุกตัวของ T01 รายงานแบบ C2 pre/post counts — พฤติกรรมที่
   Layer 0 + description กำหนด ซึ่งสอดคล้องสมมติฐาน (ข) แต่ไม่พิสูจน์)
2. **สัญญาณเดียวที่พบ: noticedLine ใน T13** — การบังคับอ่าน SKILL.md เต็มทำให้รายงาน wart
   5/5 เทียบกับ 1/5 เมื่อมีแค่ description ลอยๆ ตามกฎ C9 ที่ pre-register ไว้ = ยังสรุปไม่ได้
   · การวิเคราะห์ post-hoc (Fisher's exact, 1/5 vs 5/5) ให้ p ≈ 0.024 — **ระบุเป็น exploratory
   เท่านั้น ห้ามนับเป็นผลยืนยัน** เพราะไม่ได้ pre-register วิธีทดสอบนี้
3. **การแยก arm ดูสะอาด** — judge ตรวจแล้ว: ไม่มีรายงาน bare ที่อ้าง constants.md/ชื่อ skill;

## บทเรียนเชิงระเบียบวิธี (ปรับใช้กับรอบถัดไป)

- **กฎ spread ของ C9 ไม่ไวพอสำหรับ binary metric** — arm ที่ผลผสมกันแม้ตัวเดียวก็ทำ spread=1
  ทำให้ delta ไม่มีทางเกิน spread → รอบหน้า pre-register exact test (Fisher's) สำหรับ metric
  แบบ pass/fail (บันทึกเพิ่มใน C9 แล้ว — ใช้กับรอบถัดไปเท่านั้น ไม่ย้อนแก้คำตัดสินรอบนี้)
- **Fixture ทั้งสามง่ายเกินไปสำหรับ Opus 4.8** — ต้องมี hard-variant (เช่น race ที่ fail rate ~2%
  ให้ต้อง amplify, requirement ที่ขัดกันเองให้ต้องตีความ, wart ที่เย้ายวนกว่า dead constant)
- **In-session bare arm เป็นไปไม่ได้** — พิสูจน์แล้วด้วย probe; baseline แท้ = เปิด session ใหม่
  นอก repo ตาม `evals/README.md` §5 เท่านั้น
