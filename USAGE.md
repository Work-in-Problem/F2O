# คู่มือการใช้งาน Fable2Opus

> สอน Claude Opus 4.8 ให้ทำงานด้วย "กระบวนการ" แบบ Fable 5 — ติดตั้งครั้งเดียว ใช้ได้ทุกโปรเจกต์
> ฉบับ PDF สวยงาม: [docs/Fable2Opus-User-Guide-TH.pdf](docs/Fable2Opus-User-Guide-TH.pdf)

## โปรเจกต์นี้คืออะไร

**Fable2Opus (F2O)** คือชุด "ถ่ายทอดกระบวนการทำงาน" — เกิดจากการวิเคราะห์ว่า Claude Fable 5 (รุ่นท็อป)
ทำงานต่างจาก Claude Opus 4.8 ตรงไหน แล้วกลั่นความต่างส่วนที่เป็น**วินัยและขั้นตอน** (ไม่ใช่ความฉลาดดิบ)
ออกมาเป็นกติกา+คู่มือที่ Opus 4.8 ทำตามได้จริง ผลคือได้ Opus ที่ทำงานรอบคอบระดับใกล้เคียงรุ่นท็อป
ในราคาถูกกว่าครึ่งหนึ่งและ limit การใช้งานที่เยอะกว่า

**ทำอะไรได้บ้าง:** งานโค้ดทุกชนิดแบบมีหลักฐานทุกขั้น · ล่าบั๊กยาก/flaky ด้วยวิธีเชิงสถิติ · code review
แบบหาให้ครบก่อนกรอง · งานยาวข้าม session ไม่หล่นหาย · รายงาน/spreadsheet/slide ที่ตัวเลขตรวจทานแล้ว ·
อ่านค่าจากภาพแบบทวนสอบ · แตกงานใหญ่ให้ sub-agent ทำขนาน · hook กันคำว่า "done" ที่ไม่มีหลักฐาน

## ใช้ F2O กับไม่ใช้ ต่างกันอย่างไร

| สถานการณ์ | Opus 4.8 เพียวๆ (พฤติกรรมตั้งต้นที่ Anthropic บันทึก) | Opus 4.8 + F2O |
|---|---|---|
| บั๊กเป็นๆ หายๆ | ✗ เทสต์ผ่าน 1 รอบก็ประกาศ "fixed" | ✓ บังคับรันซ้ำ 10–30 รอบ รายงานตัวเลขก่อน/หลัง |
| คำว่า "เสร็จแล้ว" | ✗ ต้องเชื่อคำพูด | ✓ ทุก claim ชี้หลักฐานได้ · VERIFIED/NOT VERIFIED ชัด · hook บล็อกถ้าโกหก |
| คำถามระหว่างงาน | ✗ ถามเรื่องเล็กบ่อย จบด้วย "เอาไหมครับ?" | ✓ เรื่องย้อนกลับได้ตัดสินใจเอง+จดให้ veto ทีหลัง |
| ข้อเท็จจริงเวอร์ชัน/API | ✗ ตอบจากความจำ (อาจล้าสมัย) | ✓ ค้นเว็บ/อ่านไฟล์จริงก่อนพูด |
| ขอบเขตการแก้โค้ด | ✗ refactor แถม diff บวม | ✓ diff ตรงโจทย์ ปัญหาข้างเคียง "จดรายงาน" แทนแก้มั่ว |
| Review สั่ง "เอาเรื่องใหญ่" | ✗ ตกหล่นบั๊ก (recall ร่วง — Anthropic วัดเอง) | ✓ หาให้หมดก่อน ค่อยกรองตอนรายงาน |
| งานหลาย requirement | ✗ ข้อท้ายๆ หล่นหาย | ✓ checklist ครบ ปิดงานเทียบหลักฐานรายข้อ (n/n) |
| งานยาว/ทำต่อวันหลัง | ✗ ลืมบริบทหลัง context เต็ม | ✓ WORKING_NOTES.md + พิธีฟื้นบริบทจากหลักฐานจริง |

> **เพื่อความตรงไปตรงมา:** คอลัมน์ซ้ายอิงเอกสาร migration ของ Anthropic · หลักฐานเชิงเลขที่มี: instruction แบบนี้ลด ask-rate ~12 จุด (Anthropic วัด), A/B ของโปรเจกต์พบการรายงานปัญหาข้างเคียงเพิ่ม 1/5 → 5/5 เมื่อโหลด skill · สิ่งที่ให้ไม่ได้: ความฉลาดดิบ

## นี่คืออะไร (อ่าน 30 วินาที)

ชุด "กติกาการทำงาน" 3 ชั้นสำหรับ Claude Code:

| ชั้น | คืออะไร | จำเป็นไหม |
|---|---|---|
| **Layer 0** | กฎแกน 2 ไฟล์ (`CLAUDE.md` + `constants.md`) — ห้าม claim ว่าเสร็จโดยไม่มีหลักฐาน, ไม่ถามเรื่องจิ๊บจ๊อย, จดโน้ตไฟล์เดียว | ✅ จำเป็น |
| **Layer 1** | Skills 13 ตัว — คู่มือเชิงลึกที่ Claude โหลดเอง*อัตโนมัติ*ตามสถานการณ์ (แก้บั๊ก → โหลดตัว debug, จะสรุปงาน → โหลดตัวรายงาน) | ✅ จำเป็น |
| **Layer 2** | Hook ตรวจจับ — สคริปต์ที่*บล็อก*ข้อความ "เสร็จแล้ว/done" ที่ไม่มีหลักฐานจริง ก่อนถึงมือคุณ | ⭕ ทางเลือก (แนะนำ) |

**สิ่งที่จะเปลี่ยน:** Opus จะ ค้นก่อนแก้ · รัน test ซ้ำก่อนพูดว่า fixed · ทำงานจนจบเทิร์นแทนที่จะถาม "เอาไหมครับ?" · สรุปแบบ VERIFIED / NOT VERIFIED / ASSUMED
**สิ่งที่ไม่เปลี่ยน:** ความฉลาดดิบ — นี่คือวินัย ไม่ใช่สมองเสริม

## 1. สิ่งที่ต้องมีก่อน

- **Claude Code** (แอป desktop หรือ CLI) ที่ล็อกอินใช้งานได้แล้ว
- **โมเดล Opus 4.8** — ใน session พิมพ์ `/model` แล้วเลือก Opus (ชุดนี้ออกแบบเพื่อ Opus 4.8 แต่ใช้กับโมเดลอื่นก็ไม่พัง)
- **Python 3** — เฉพาะถ้าจะติดตั้ง Layer 2 (macOS มีอยู่แล้ว ตรวจด้วย `python3 --version`)

ข้อมูลชุดนี้อยู่ที่ GitHub: **https://github.com/Work-in-Problem/F2O** — ติดตั้งแบบคำสั่งเดียวได้ 2 ช่องทาง (ดูกล่องด้านล่าง) หรือ clone มาติดตั้งมือก็ได้

## 2. ติดตั้งแบบคำสั่งเดียว (แนะนำ)

**ช่องทาง A — Claude Code plugin (ครบทุกชั้น: skills + กฎแกนอัตโนมัติ + hook):**
```
claude plugin marketplace add Work-in-Problem/F2O
claude plugin install fable2opus@f2o
```
(หรือพิมพ์ `/plugin` ในแอปแล้วเลือก marketplace `Work-in-Problem/F2O`) — เปิดใช้แล้วกฎ Layer 0
จะถูกฉีดเข้า**ทุก** session อัตโนมัติ ปิดชั่วคราวได้ด้วย disable plugin หรือ `FABLE2OPUS_HOOKS_OFF=1`

**ช่องทาง B — npx skills (ได้เฉพาะ skills 13 ตัว):**
```
npx skills add Work-in-Problem/F2O --all -y -g
```
ช่องทางนี้*ไม่พก*กฎแกน (Layer 0) กับ hook — ต้อง copy เองตามขั้น 2–3 ของหัวข้อถัดไป

## 2.5 ติดตั้งมือจาก clone (ทางเลือก, ~2 นาที)

เปิด Terminal แล้วรันทีละบรรทัด (ถ้าโฟลเดอร์ต้นทางอยู่ที่อื่น แก้บรรทัดแรกให้ตรง):

```sh
git clone https://github.com/Work-in-Problem/F2O.git && SRC="$PWD/F2O"

# ขั้น 1 — ติดตั้ง skills 13 ตัวแบบ global (ใช้ได้ทุกโปรเจกต์บนเครื่องนี้)
mkdir -p ~/.claude/skills
cp -R "$SRC/skills/." ~/.claude/skills/

# ตรวจว่าเข้าครบ (ต้องเห็นชื่อ 13 โฟลเดอร์ เช่น verifying-before-claiming, finishing-the-turn ...)
ls ~/.claude/skills
```

```sh
# ขั้น 2 — ติดตั้ง Layer 0 ลง "โปรเจกต์ที่จะใช้งาน" (ทำซ้ำกับทุกโปรเจกต์ที่ต้องการ)
cd /path/to/your-project        # ← แก้เป็นโปรเจกต์จริงของคุณ

cp "$SRC/core/constants.md" ./constants.md

# ถ้าโปรเจกต์ยังไม่มี CLAUDE.md:
cp "$SRC/core/CLAUDE-core.md" ./CLAUDE.md
# ถ้ามี CLAUDE.md อยู่แล้ว: เอาเนื้อหาไปต่อท้ายแทน
#   cat "$SRC/core/CLAUDE-core.md" >> ./CLAUDE.md
```

```sh
# ขั้น 3 (ทางเลือก) — ติดตั้ง hook กันหลอกว่าเสร็จ ลงโปรเจกต์เดียวกัน
mkdir -p .claude/hooks
cp "$SRC/hooks/claim_gate.py" .claude/hooks/
cat > .claude/settings.json <<'JSON'
{"hooks":{"Stop":[{"hooks":[{"type":"command","command":"python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/claim_gate.py","timeout":30}]}]}}
JSON
# ⚠️ ถ้าโปรเจกต์มี .claude/settings.json อยู่แล้ว อย่า copy ทับ —
#    ให้เปิดไฟล์แล้วเพิ่มเฉพาะก้อน "hooks" นี้เข้าไป:
#    {"hooks":{"Stop":[{"hooks":[{"type":"command","command":"python3",
#      "args":["${CLAUDE_PROJECT_DIR}/.claude/hooks/claim_gate.py"],"timeout":30}]}]}}
```

> **หมายเหตุเรื่อง CLAUDE.md ของ Fable2Opus:** ท้ายไฟล์มีตาราง "Skill router" ชี้ไปยัง skill ทั้ง 13 — ใช้ได้เลยไม่ต้องแก้ เพราะชื่อ skill ตรงกับที่ติดตั้ง global ในขั้น 1

## 3. เรียกใช้ยังไง

**คำตอบสั้น: ไม่ต้องเรียก — มันทำงานเอง**

1. `cd` เข้าโปรเจกต์ที่ติดตั้ง Layer 0 แล้ว → เปิด `claude` → `/model` เลือก **Opus 4.8**
2. สั่งงานตามปกติ Claude จะโหลด skill ที่ตรงสถานการณ์เองอัตโนมัติ เช่น:
   - "แก้บั๊ก login แล้ว test ให้ผ่าน" → โหลด `root-causing-bugs` + `verifying-before-claiming`
   - "รีวิว PR นี้หน่อย" → โหลด `reviewing-code`
   - "ทำรายงานยอดขายจาก csv นี้" → โหลด `producing-deliverables`
3. อยากเรียกเองเจาะจงก็ได้: พิมพ์ `/` แล้วเลือกชื่อ skill (เช่น `/verifying-before-claiming`) หรือพิมพ์บอกตรงๆ ว่า "ใช้ skill scoping-code-changes ด้วย"

**เคล็ดให้ได้ผลสูงสุดกับ Opus 4.8** (จากคู่มือทางการของ Anthropic):
- ให้ **spec ครบตั้งแต่ข้อความแรก** — Opus ทำงานยาวได้ดีที่สุดเมื่อได้โจทย์เต็มในเทิร์นเดียว อย่าป้อนทีละนิด
- ปล่อยให้ทำจนจบ — ไม่ต้องจ้ำจี้ระหว่างทาง กติกา finishing-the-turn คุมให้มันไม่หยุดกลางคันอยู่แล้ว

## 4. รู้ว่าติดตั้งสำเร็จจากอะไร

เปิด session ใหม่ในโปรเจกต์แล้วลองสั่ง: *"แก้ typo ในไฟล์ X แล้วบอกด้วยว่า verify ยังไง"* — สัญญาณที่ควรเห็น:

- ✅ มันอ่านไฟล์ก่อนแก้ (ไม่แก้มั่ว) และ**ไม่ทำพิธีใหญ่โต**กับงานจิ๋ว (กติกา proportionality)
- ✅ สรุปปิดงานขึ้นต้นด้วยผลลัพธ์ 1 บรรทัด + มีหมวด VERIFIED พร้อมคำสั่งที่รันจริง
- ✅ ไม่จบเทิร์นด้วย "Want me to also…?" สำหรับงานที่ควรทำต่อเอง
- ✅ (ถ้าติดตั้ง hook) ลองสังเกตว่าเมื่อไหร่ที่มันพูดว่า done โดยไม่ได้รันอะไร จะโดนบังคับให้กลับไป verify ก่อน

หรือเช็คแบบกลไก: `ls ~/.claude/skills | wc -l` ต้องได้ **13**

## 5. ปัญหาพบบ่อย

| อาการ | ทางแก้ |
|---|---|
| Skill ไม่โหลด / ไม่เห็นใน `/` | ปิด-เปิด session ใหม่ (skills โหลดตอนเริ่ม session) · ตรวจ `ls ~/.claude/skills` |
| Hook บล็อกทั้งที่ไม่ควร | ปิดชั่วคราว: `export FABLE2OPUS_HOOKS_OFF=1` แล้วเปิด session ใหม่ · เจอเคสแปลกแจ้งเก็บไว้ปรับ hook |
| ทำพิธีเยอะเกินกับงานเล็ก | บอกตรงๆ ได้เลย ("งานนี้เล็ก ข้ามพิธี") — ทุกกฎมี escape hatch ให้ข้ามพร้อมเหตุผล 1 บรรทัดอยู่แล้ว |
| ใช้ token มากขึ้น | ปกติ — งาน verify หนักกิน ~1.3–2 เท่า แลกกับความน่าเชื่อถือ; งานเล็กแทบไม่ต่าง |
| เลขในกฎขัดกัน | `constants.md` ชนะเสมอ (กติกาข้อ 4 ของ Layer 0) |

## 6. อัพเดต / ถอนการติดตั้ง

- **อัพเดต:** แก้ที่ต้นทาง (`$SRC`) แล้วรันคำสั่ง copy ขั้น 1–3 ซ้ำ (ทับของเก่า)
- **ถอน skills:** ลบโฟลเดอร์ที่ไม่ต้องการออกจาก `~/.claude/skills/` (เลือกลบรายตัวได้)
- **ถอน Layer 0:** ลบ `constants.md` และส่วน Fable2Opus ใน `CLAUDE.md` ของโปรเจกต์
- **ถอน hook:** ลบก้อน `hooks` ใน `.claude/settings.json` (หรือลบไฟล์ถ้าสร้างใหม่ทั้งไฟล์) + ลบ `.claude/hooks/`

## 7. Skills ทั้ง 13 ตัว — ใครทำอะไร โหลดเมื่อไหร่

| Skill | ทำอะไร | โหลดเองเมื่อ |
|---|---|---|
| `verifying-before-claiming` | พิสูจน์ก่อนพูดว่าเสร็จ: repro บั๊กก่อนแก้, รัน flow จริง, อ่าน output ไม่ใช่ exit code | ทำ feature/แก้บั๊ก/จะพูดว่า done |
| `finishing-the-turn` | ไม่ถามเรื่องเล็ก ทำงานจนจบเทิร์น ห้ามจบด้วยคำสัญญา | ก่อนถามคำถาม/จบเทิร์น/เจอ error |
| `search-first-context` | ค้น/อ่านก่อนพูดหรือแก้ — เวอร์ชัน API ราคา ต้องเช็คก่อน ไม่เดาจากความจำ | เริ่มงานใน codebase / จะพูด fact |
| `planning-to-done` | แตกโจทย์เป็น checklist ครบทุกข้อก่อนลงมือ ปิดงานด้วยหลักฐานรายข้อ | งานหลายส่วน/feature/migration |
| `scoping-code-changes` | diff เล็กตรงโจทย์ ไม่ refactor แถม เจอปัญหาข้างเคียงให้จดไม่ให้แก้ | ก่อนแก้โค้ดใดๆ |
| `root-causing-bugs` | debug แบบตั้งสมมติฐาน-ทดลอง หา root cause ไม่แปะ band-aid, จัดการ flaky แบบสถิติ | มีบั๊ก/test พัง/อาการ "เป็นบางครั้ง" |
| `reviewing-code` | รีวิวแบบหาให้หมดก่อนแล้วค่อยกรอง + ตรวจทุก finding กับโค้ดจริง | รีวิว PR/diff/หา bug ในโค้ด |
| `delegating-parallel-work` | แตกงานใหญ่ให้ sub-agent ทำขนาน แล้วตรวจรวมเอง | งาน 3+ ส่วนอิสระ/หลาย package |
| `managing-working-memory` | จดโน้ต WORKING_NOTES.md ให้งานยาวรอดข้าม session/compaction | งานยาว/ทำต่อพรุ่งนี้/หลัง compact |
| `outcome-first-reporting` | สรุปแบบผลลัพธ์ขึ้นก่อน มีหมวด "ยังไม่เสร็จ" ชัดๆ ไม่เล่านิทาน | ตอนเขียนสรุป/ตอบสถานะ |
| `producing-deliverables` | ทำรายงาน/spreadsheet/slide แบบตรวจข้อมูลก่อนคำนวณ + เปิดไฟล์ทวนก่อนส่ง | งานที่ผลลัพธ์เป็นเอกสาร ไม่ใช่โค้ด |
| `extracting-from-images` | อ่านค่าจากภาพ/สแกนแบบ crop-ซูม-ทวนสอบ ไม่เดาตัวเลขที่มองไม่ชัด | ต้องดึงตัวเลข/ตารางจากรูป |
| `inventorying-capabilities` | สำรวจเครื่องมือ (MCP/skills) ที่มีจริงก่อนบอกว่า "ทำไม่ได้" | session ที่ต่อเครื่องมือพิเศษ |
