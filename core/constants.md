# Fable2Opus — Canonical Constants

Single source of truth for every numeric threshold in the curriculum. Skills cite this
table; they never restate different numbers. If any skill text, example, or memory entry
conflicts with this table, **this table wins**.

| # | Constant | Value | Notes |
|---|---|---|---|
| C1 | Flake re-run count | **10 consecutive runs**; **30 runs** when the observed failure rate is under ~20% | One green run of a flaky test is evidence of nothing. 10/10 green only bounds the failure rate below ~26% at 95% confidence. |
| C2 | Statistical "fixed" claim (nondeterministic bugs) | Post-fix failures = 0 across **M = max(20, min(100, ceil(3 / baseline-failure-rate)))** runs | Report as "pre-fix k/N failures, post-fix 0/M". If M is impractical — including whenever the 100-run cap engages (baseline under ~3%): 0/100 there is weak evidence, with false-green odds reaching ~37% at a 1% rate — the verdict is "high confidence, not proven", never "fixed"; prefer amplifying the failure rate so a computable M exists. |
| C3 | Working-notes file trigger | Task will plausibly exceed **~25 tool calls**, spans multiple components/repos, or is explicitly long-running / multi-session | One file only: `WORKING_NOTES.md` (see CLAUDE.md §3). |
| C4 | Fan-out (sub-agent) trigger | **3+ genuinely independent workstreams, each ≥ ~10 minutes of agent work** | Never fan out on file count alone. Below the bar, do the work yourself. |
| C5 | Error escalation | After **3 materially different** fix attempts | "Materially different" = different tool, different layer, or different hypothesis — re-running the same command or permuting flags does not count. |
| C6 | Planning right-sizing | Skip planning ceremony when the task is completable in a handful of calls touching 1-2 files AND has no multi-part requirements | A principle with examples, not a hard number — a genuinely tricky 4-call task still deserves a plan. |
| C7 | Memory-file size budget | Consolidation pass when the durable memory file exceeds **~150 lines** | Merge duplicates, delete superseded entries, keep trigger→action form. |
| C8 | Re-grounding trigger | When the user plausibly lost the thread — e.g. their first message after **~15+ tool calls** since your last summary | The number is an example, not a gate. |
| C9 | Eval minimum sample size | **N = 5 runs per arm/condition** minimum | Below this, draw no conclusion. Always report mean + min–max spread per arm; a delta within the spread is "no measurable difference". For binary pass/fail metrics the spread rule is insensitive (one mixed arm forces spread = 1) — pre-register an exact test (e.g. Fisher's) for those instead. Adopted 2026-07-10 after A/B run 1; applies to future runs only, never retroactively. |
| C10 | Fast full-suite bar | Full test suite finishing in **~2 minutes or less** → run all of it after changing shared code | Example-shaped like C6/C8, not a hard gate — a 3-minute suite on a wide-blast-radius change is still worth running. |
| C11 | Very-large enumeration handling | Batch size **~25–50 hits per batch**; treat **100+ items** as the scope-doubt trigger | Both numbers are examples, not gates; the batching + checkpoint procedure lives in `finishing-the-turn` rule 6. |
| C12 | Sub-task sizing band (delegation) | One coherent deliverable with a runnable done-condition: roughly **10–30 minutes** of agent work; proxy **~1–10 files** or one module/test suite | Floor inherits C4's per-stream floor. The file-count proxy is a proxy, not the definition — split along module/directory/test-suite ownership boundaries and count deliverables, never files. |
| C13 | Survey-delegation scale | Wide read on the order of **~a dozen files**; each survey-agent digest capped at **~30 lines** | Both numbers are examples, not gates (C6/C8-shaped). The rule is two-sided: reading ~12 small files yourself is often cheaper and higher-fidelity than 12 sub-agent digests — delegate on context-budget threat or cleanly parallel questions, never on file count alone. |
| C14 | Whole-file read cutoff | Under **~400 lines** → read the ENTIRE target file before the first symbol/behavior-changing edit; above → the enclosing function/class plus the imports and exports sections | Curriculum-wide: any skill prescribing pre-edit or pre-summary reading cites this row (currently `search-first-context` rule 2). |
| C15 | Breadth-first opening window | First **~3–5 tool calls** on a new repo task stay wide and cheap (ls/Glob of candidate dirs, Grep of key domain terms, Read of README/CLAUDE.md/manifest) | Example-shaped like C6/C8, not a hard gate. |
| C16 | Staleness re-read trigger (call-count arm) | Re-read a region before editing it again when **~20+ tool calls** have elapsed since your last read of it | The number is an example, not a gate (like C8). The non-numeric arms — a formatter/hook ran; the user or another agent touched the repo — always apply regardless of count. |
| C17 | Abstraction extraction gate (rule of three) | Extract a shared helper, base class, interface, factory, registry, or options/config object only at **3+ concrete usages**; below that, duplicate the concrete code | Hard gate with an in-rule justified-exception valve stated where applied (currently `scoping-code-changes` rule 5). An explicit task request for the abstraction also satisfies the gate. |
| C18 | Right-sized test count for a code change | Fewest tests that pin the changed observable behavior — typically **1–3** | Example-shaped like C6/C8, not a hard gate. Select per `verifying-before-claiming` rule 15: enumerate candidate cases fully, then filter — "fewest" governs what you write, never what you consider. |
